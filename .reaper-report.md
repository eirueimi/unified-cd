# Approval Timeout Reaper — Implementation Report

## Status: DONE

## What was implemented

A controller-side, leader-elected periodic reaper that marks expired `Pending`
approvals as `TimedOut` in the `run_approvals` audit table. It only touches the
audit row — it never changes run status (the agent already fails the step on
timeout → run Failed).

### Files changed
- `internal/store/store.go` — added `MarkExpiredApprovalsTimedOut(ctx) (int, error)`
  to the `Store` interface, near the other approval methods.
- `internal/store/postgres.go` — implemented `MarkExpiredApprovalsTimedOut` with a
  single `UPDATE ... SET status='TimedOut', decided_by='system', decided_at=now()
  WHERE status='Pending' AND timeout_at IS NOT NULL AND timeout_at < now()`.
- `internal/controller/approval_reaper.go` (new) — `RunApprovalReaper` ticker loop
  and `runApprovalReaperAsLeader` helper, mirroring `RunCacheCleanup`/
  `runCacheCleanupAsLeader` (ticker → `AcquireAdvisoryLock` → early-return when
  `release == nil` → `defer release()` → `MarkExpiredApprovalsTimedOut`).
- `cmd/controller/main.go` — registered `go controller.RunApprovalReaper(ctx, pg, time.Minute)`
  alongside the other periodic launches (unconditional; does not require an object store).
- `docs/jobs.md` — replaced the stale "v1 limitation: audit row stays Pending" note
  with the truthful behavior (reaper marks it TimedOut / decided_by=system within ~1m;
  run still fails via the agent).

### Tests (TDD — written failing first, confirmed red via `go vet`, then green)
- `internal/store/postgres_approvals_test.go` — appended `TestPostgres_MarkExpiredApprovalsTimedOut`:
  expired Pending → TimedOut + decided_by=system + decided_at set; future Pending untouched;
  already-Approved untouched; idempotent second reap returns 0.
- `internal/controller/approval_reaper_test.go` (new) — `TestRunApprovalReaperAsLeader_MarksExpired`
  calls `runApprovalReaperAsLeader` directly against a real store. The full ticker loop is
  intentionally NOT tested — the leader helper covers all the logic.

All ran against a real Postgres via dockertest (`store.NewTestPostgres`).
